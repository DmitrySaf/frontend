# Bean — DESIGN.md

> Снято с живого кода (app/globals.css + src/shared/components). Актуально на этап 17 (дизайн-ревью, в процессе): включает тёмную тему, самохостинг Onest и удаление `bean-design-system`.
> Источник истины — сам код; этот файл — карта для агентов и ревью.

## 1. Visual Theme

Светлая продуктовая тема Apple/Whop-школы: белые поверхности, тонкие серые рамки-волоски, крупные радиусы, один синий акцент. Плотность — комфортная продуктовая (не маркетинговый воздух и не кокпит). Светлая тема — база; тёмная реализована ретрофитом через переворот токенов (см. §2).

## 2. Color

Токены объявлены в `app/globals.css` (@theme → CSS-переменные, HSL-триплеты). Шкалы тональные, выверены по извлечённым токенам apple.com/whop.com.

| Роль | Токен | Значение | Использование |
|---|---|---|---|
| Фон приложения | `--color-background` | белый | страницы, карточки |
| Каркас/рельсы | `gray-100` | hsl(0 0% 96%) | фон рейла/drawer, треки, ховеры ghost |
| Рамки | `gray-200` | hsl(0 0% 90%) | борта карточек, инпутов, разделители |
| Активные строки | `gray-200/70` + ring `gray-300/70` | — | выбранный канал/таб/урок |
| Вторичный текст | `gray-500` / `gray-600` | 55% / 45% | метаданные / подписи (AA на белом) |
| Поверхность | `--color-surface` | white / dark 0.15 | карточки/бары/поля (`bg-surface`, переворачивается в тёмной) |
| Основной текст | `--color-ink` | oklch(0.193) / dark 0.96 | body (`text-ink`, переворачивается в тёмной) |
| Акцент | `primary-500` | hsl(210 100% 45%) ≈ #0074E6 | кнопки, ссылки-действия, фокус, селекция |
| Акцент-ховер | `primary-400` | 49% | hover primary-кнопки (светлеет, как apple.com) |
| Акцент-нажатие | `primary-600` | 40% | active; также текст-ссылки на белом |
| Светлый тинт | `primary-50` | 97% | фон выбранной опции (тип таба, тариф) |
| Заливка | `--color-fill` / `-hover` | 0.962 / dark 0.26 | серая кнопка «Отмена», кружок закрытия (Apple systemFill) |
| Скрим | `--color-scrim` | black/18% / dark 60% | подложка модалки — светлая, работу делает blur |
| Опасность | `--color-danger` | oklch(0.59 0.196 36) | удаление, выход, ошибки |
| Бренд-лайм | `--color-brand` | oklch(0.9 0.235 128) = #B0F900 | **только айдентика**, см. ниже |

Правила: один акцент; цвет = состояние или действие, не украшение; серый текст на цветном фоне запрещён (использовать тёмный той же гаммы/прозрачность белого); графики — primary-500 для текущего/линий, gray-100+рамка gray-300/70 для фоновых баров.

### Бренд-лайм — цвет принадлежности, а не смысла

`#B0F900` — кислотный лайм: стоит вплотную к границе гаммы sRGB (насыщенность 0.235 против 0.106 у прежнего #DEFCAD, который был пастелью и терялся среди аватарок сообществ). Тон 128° унаследован от старого лайма (125°), поэтому знак узнаётся.

**Главное правило: лайм — это заливка, а не чернила.**

| Сценарий | Контраст | |
|---|---|---|
| Тёмный текст/знак **на** лайме | 14.7:1 | можно на любом фоне |
| Лайм текстом/иконкой на **тёмном** | 15.1:1 | можно |
| Лайм текстом/иконкой/линией на **светлом** | 1.28:1 | **запрещено** |
| Край лаймовой плашки на белом | 1.28:1 | форма не читается — мелких лаймовых кнопок на белом не бывает |

**Где живёт:** фавикон (`app/icon.svg`), OG-картинки, лендинг, маркетинг, мерч. На тёмном фоне лайму можно всё; на светлом — только крупной плашкой с тёмным контентом сверху.

**Где НЕ живёт:** success, «урок пройден», «оплачено», прогресс, статусы, бейджи, любая кнопка внутри приложения. Если лайм начнёт означать «успешно», он перестанет означать «Bean» — и обратное: цвет для success нужен свой, тихий, из полосы баланса (L 0.55–0.65), заметно дальше по тону. Синий остаётся единственным цветом действия.

### Версии логотипа

Все версии объявлены **в одном месте** — `LOGO_TILE_VARIANTS` и `LOGO_TONES` в [Logo.tsx](src/shared/components/Logo.tsx). Не собирать плитку из классов вручную: выбирать вариант.

**Плитка (`LogoTile variant=…`)** — несёт СВОЮ подложку, а не цвет страницы, поэтому одинакова в обеих темах.

| Вариант | Подложка | Знак | Знак/фон | Где |
|---|---|---|---|---|
| `acid-on-black` *(дефолт)* | #0B0D0A | #B0F900 | 15.2:1 | продукт, фавикон `app/icon.svg` |
| `white-on-black` | #0B0D0A | белый | 19.5:1 | когда бренд-цвет неуместен |
| `black-on-white` | белый | #0B0D0A | 19.5:1 | печать, светлые документы |
| `black-on-acid` | #B0F900 | #0B0D0A | 15.2:1 | маркетинг — самый громкий, **не для рейла** |

**Обводка входит в вариант и не опциональна.** На «своём» фоне край плитки пропадает: чёрная на тёмном рейле — 1.29:1, белая на светлом — 1.09:1, зелёная на светлом — 1.17:1. Без хайрлайна (`ring-white/10` для тёмных подложек, `ring-black/10` для светлых) плитка перестаёт быть плиткой. Сам знак читается всегда.

**Тон локапа (`LogoLockup tone=…`)** — знак и слово «Bean» красятся заодно. Голым текстом фон не свободен:

| Тон | На светлом | На тёмном |
|---|---|---|
| `ink` *(дефолт)* | ок | ок — токен переворачивается с темой |
| `black` | 19.5:1 ок | 1.02:1 **запрещено** |
| `white` | 1.00:1 **запрещено** | 19.2:1 ок |
| `brand` | 1.28:1 **запрещено** | 15.0:1 ок |

Сомневаетесь — берите `ink`: он единственный безопасен на обоих фонах.

**Почему плитка не путается с сообществами.** Их аватарки — тот же `size-12` и тот же радиус 14px (плитка обязана совпадать с `Avatar size="l"`, иначе в рейле углы разъезжаются), но заливка **светлым пастельным** (насыщенность ~0.023 против 0.235 у бренда). Прежний `#DEFCAD` был сам членом пастельного семейства и потому читался как ещё одно сообщество. Плюс хайрлайн-разделитель под знаком в рейле — паттерн Discord.

**Тёмная тема** (next-themes, `attribute="class"` → `.dark` на `<html>`, тумблер в рейле `MainSidebar`). Ретрофит через переворот значений токенов в `.dark`, без `dark:`-утилит: `surface`/`ink` меняются местами, серая шкала инвертирует роль (50-300 — тёмные поверхности, 500-900 — светлый текст), primary светлеет для контраста на тёмном. Литеральными остаются `white`/`black` — текст-на-цвете (кнопки, `text-white`); скрим модалки вынесен в собственный переворачиваемый токен `--color-scrim`; видимые чёрные акценты (индикатор активного сообщества, активный таб настроек) используют `bg-ink`, чтобы переворачиваться.

## 3. Typography

- Семейства: `--font-sans` = **Onest**, самохостинг через `@font-face` в [globals.css](app/globals.css) (`public/fonts/onest-{latin,latin-ext,cyrillic}.woff2`, variable — один файл на веса 100–900) — основной UI/боди-шрифт с первоклассной кириллицей. `next/font/google` не используется: сеть окружения флапала и шрифт не скачивался, а `@theme` плодил хрупкую индирекцию `var(--font-sans)→var(--font-onest)` — семейство подключено прямым именем.
- `--font-mono` = **системный моно-стек** (`ui-monospace`, SF Mono, Menlo, Consolas) — только цифры/деньги/тайминги. Geist Mono убран (лишний вес ради латиницы, которой в цифрах не видно). `<html lang="ru">`.
- База: 16px / line-height 1.5; `--text-sm` 14px/1.43 — списки, вторичные строки; заголовки страниц 24px bold, секций 15–17px bold, модалок `text-2xl` + `tracking-tight`.
- Моно — только цифры/деньги/тайминги/OTP (`font-mono`): статистика, `мм:сс`, счётчики.
- Микротекст 11–13px используется плотно (метки каналов, подписи) — контраст держать ≥4.5:1 (gray-500+).
- Копирайтинг: русский, sentence case, на «вы», без эмодзи; деньги «₽ 990 / мес» с неразрывными пробелами.

## 4. Components (src/shared/components)

**Движок — гибрид, тема всегда наша.** Кит стоит на двух движках, и оба видны только внутри
`src/shared/components/**` (граница защищена линтом — `noRestrictedImports` в `biome.json`;
снаружи приложение знает лишь баррель `@/shared/components`):

- **Примитивы (bucket A) — чистый `react-aria-components` (RAC)** либо вовсе без движка:
  `Button`, `Switch`, `Tabs`, `SegmentedControl`, `Tooltip`, `Dropdown` — на RAC (headless:
  поведение/доступность, ноль пикселей); `Input`, `Textarea`, `Avatar`, `Separator`, `Skeleton`
  движок не держат вовсе (RHF + нативный элемент / `<img>` / `<div>`). Оформление — целиком наши
  классы и токены на том же DOM.
- **Тяжёлое поведение (bucket B) — остаётся на HeroUI v3** (`@heroui/react` на React Aria):
  `Dialog` (Modal+Sheet, dual-tree + фокус-менеджмент), боковой `Drawer` в `CommunityShell`,
  `Toast` (очередь + императивный API). Пересобирать это на RAC не окупается — владелец разберёт
  точечно позже.

**Движок — деталь реализации, гасить у RAC нечего:** нет дефолтной заливки, а голую `<button>`
сбрасывает Tailwind preflight. Нейтрализаторы HeroUI-хрома (`.button--primary`, `.input`)
остаются актуальны **только для bucket B**. Состояния (hover/press/focus/selected/disabled) у RAC —
через `data-*` (`data-[selected]`, `data-[pressed]`, `data-[focus-visible]`); у полиморфных
(Button — `<button>`/`<a>`) — CSS-псевдоклассы (якорь RAC data-* не отдаёт). Варианты — один
движок, **cva** (`tailwind-variants` удалён); `cn()` = clsx + tailwind-merge разруливает конфликты.

Внутри коллекций (`Menu`/`ListBox`) все дети — collection-компоненты (`Item`/`Header`/`Separator`);
триггеры дропдаунов/тултипов цепляются на существующий элемент через `Pressable`/`Focusable`
(cloneElement, без лишнего DOM), а не через свой `*.Trigger`.

Тиры токенов — §9; уровни компонентов (Atomic ↔ FSD) — §10.

| Компонент | Ключевые решения |
|---|---|
| `Button` | RAC Button (headless) + `next/link` для href; 7 тем, **4 ступени (sm/md/lg/xl = 32/36/40/48)**, `shape` = rounded/pill, иконки пропами `Icon`/`IconRight`; `active:scale-[0.98]`, focus-ring `primary-500/45`, transition 150ms `ease-out-quart`; `onPress` под нашим `onClick`, `isPending` под `isLoading`. | `SegmentedControl` | RAC Tabs (`TabList`/`Tab`; имя и терминология «сегмент-контрол» сохранены); трек gray-100, активный сегмент — пилюля **без обводки, только тень** (`data-[selected]`, без слайд-индикатора; реш. №10); радиусы **концентричны** (см. §4.3) |

### 4.1 Закрытие модалок

Раскладка кнопок и наличие крестика выводятся из трёх осей: **что теряется** при случайном закрытии, **равноценны ли** варианты выбора, и **достижимость** (на телефоне верхний правый угол — худшая точка для пальца, низ — лучшая).

**Каталог раскладок.** Полный список известных паттернов — включая те, что мы сейчас не применяем: если понадобится новый тип окна, выбирать из этого списка, а не изобретать.

| | Раскладка | Референс | Статус у нас |
|---|---|---|---|
| **P1** | Пара кнопок справа внизу, крестика нет | macOS-диалоги, Material 3 basic dialog | подтверждения (десктоп) |
| **P2** | Пара справа внизу + крестик | Stripe, GitHub Primer, Shopify Polaris | формы (десктоп) |
| **P3** | 50/50 на всю ширину нижней панели | iOS-алерты, Fluent ContentDialog, Vercel | подтверждения (мобильный), `layout="split"` |
| **P4** | Одна кнопка на всю ширину + крестик, «Отмены» нет | оплата/логин/пейволы у Apple в вебе, Whop | `CheckoutModal` |
| **P5** | Стопка на всю ширину: действие сверху, отход призраком под ним | мобильные шиты, Material full-screen | формы (мобильный), `layout="stack"` |
| **P6** | Верхняя панель: «Отмена»/✕ слева, действие текстом справа, футера нет | iOS page sheet, Material full-screen dialog | **не используется** — кандидат для полноэкранных мобильных форм |
| **P7** | Только крестик, кнопок действий нет | информационные окна | `InviteDialog`, `AuthRequiredDialog` |

Отдельно: **свайп вниз** — не раскладка кнопок, а замена клику снаружи на мобильном. Именно им Apple компенсирует отсутствие крестика в шите. **Не реализован**; нужен, если модалки на `<768px` будут переведены в bottom sheet.

Если крестик на мобильном всё же нужен (P6) — он ставится **слева**: правый верхний угол недостижим большим пальцем на крупном экране.

| Тип окна | Десктоп | Мобильный | Крестик | Клик снаружи | Кто |
|---|---|---|---|---|---|
| Подтверждение (необратимое) | пара справа | 50/50 (`layout="split"`) | нет | **нет** | `DeleteDialog` |
| Форма / создание | крестик + пара справа | стопка (`layout="stack"`) | да | только если `!isDirty` | `Community/Channel/Tier/AddCard` |
| Одно действие (конверсия) | крестик + кнопка на всю ширину | то же | да | да | `CheckoutModal` |
| Информационное | только крестик | то же | да | да | `InviteDialog`, `AuthRequiredDialog` |

Правила, из которых это следует:

- **Один подписанный выход на окно.** Есть футер с «Отменой» → крестика нет (иначе неподписанный контрол дублирует подписанный). Нет футера → крестик и есть выход.
- **50/50 — только для симметричного выбора.** В форме «Отмена» и «Создать» не равноценны: одна — цель, вторая — аварийный выход. Дать им по половине ширины — соврать про вес. Форма на мобильном = стопка: действие сверху, отход под ним.
- **В подтверждении акцент (плотная заливка) — у безопасного варианта.** Деструктив — `destructiveTonal`, а не сплошной `destructive`: залитый красным блок на пол-экрана превращает необратимое действие в самую крупную мишень окна. Красный обязан читаться как опасность, а не как главный акцент (логика iOS-алерта).
- **Клик снаружи и Esc — ускорители, а не выход.** Разрешены ровно там, где закрытие ничего не стоит. Крестик в подтверждении опасен не тем, что можно случайно удалить (случайное закрытие как раз безопасно), а тем, что даёт закрыть окно **не прочитав вопрос**.
- **Футер липкий.** Иначе на высокой модалке «Отмена» уезжает за скролл, и единственным выходом остаётся крестик в самом недостижимом углу. Контент растворяется под футером градиентом (`::before`), а не обрывается по жёсткой линии.

Паттерны: селекция опций = рамка `primary-500` + фон `primary-50` (радио/плитки/тарифы); строки списков (каналы/уроки/табы) = r10, hover `gray-200/45`, active `gray-200/70` + inset-ring `gray-300/70`; «···»-меню появляются по hover на десктопе и видимы на таче.

### 4.2 Кнопки

**Ступень — это не высота, а согласованный набор:** вместе с ней едут кегль, трекинг, радиус и размер иконки. Раньше менялись только высота и кегль, а вес и трекинг стояли колом на всех размерах — отсюда ощущение, что текст не в размер кнопки.

**Сетка — 4 ступени `sm/md/lg/xl` = 32/36/40/48.** Прежние 28px и 44px убраны (реш. №3):
ступень 40px (`lg`) — новая, заняла роль тач-дефолта/основной CTA. Высоты драйвим своими
классами по сетке 4px (движок RAC headless и размеров не навязывает); радиус/кегль —
лестничные токены `--radius-control-*` / `--text-btn-*`.

| size | Высота | Кегль | Вес | Трекинг | Паддинг | Радиус | Иконка | Роль |
|---|---|---|---|---|---|---|---|---|
| `sm` | 32 | 13 | 600 | +0.01em | 12 | 10 | 16 | «···»-меню, компактные тулбары, плотные ряды |
| `md` | 36 | 14 | 600 | 0 | 14 | 10 | 16 | десктопный дефолт: настройки, карточки, админка |
| `lg` | 40 | 16 | 600 | −0.01em | 16 | 12 | 18 | тач-дефолт / основная CTA: футер модалки, формы |
| `xl` | 48 | 17 | 600 | −0.01em | 24 | 14 | 20 | главный CTA: логин, оплата, витрина |

Три правила, по которым выведены числа:

1. **Кегль ≈ 0.36–0.40 высоты.**
2. **Радиус растёт со ступенью, держа постоянное отношение ≈0.28–0.31 к высоте.** Это и есть «эппловская» форма: одинаковая оптическая округлость на всех размерах, а не одинаковый радиус — фиксированные 12px на кнопке 32px выглядят пузырём, а на 48px остро. Whop (`frosted-ui`) держит то же правило при ≈0.25.
3. **Вес — константа 600.** Системная кнопка iOS — 17pt Semibold; по размерам вес не гоняет никто из референсов. Единственная оптическая поправка — **трекинг, и он идёт против размера**: у Onest нет оптических начертаний (SF Text / SF Display), поэтому разницу плотности компенсируем межбуквенным интервалом вручную.

`xl` = 48px остаётся в шкале как **ячейка рейла** — аватарки сообществ те же 48px, и кнопка «+» обязана в неё попадать.

**Темы (7).** `primary` · `secondary` (серая заливка — пара к primary в футере) · `outline` · `ghost` · `destructive` (сплошной; вне пары 50/50) · `destructiveTonal` (подтверждения, см. §4.1) · `destructiveGhost` (второстепенное удаление в общем ряду — «Удалить урок» рядом с «Сохранить»: постоянная заливка там слишком громкая, фон приходит только на ховере).

**Форма.** `shape` = `rounded` (дефолт, радиус ступени) | `pill` (полное скругление — витрина, лендинг, пейвол). Отдельного `circle` нет и не нужно: `iconOnly` уже квадратит кнопку (`aspect-square`), поэтому `pill` на ней даёт ровный круг.

**Иконки — только пропами** `Icon` (слева) и `IconRight` (справа): размер берётся от ступени. Класть иконку в `children` с ручным `size-*` **запрещено** — именно так в коде развелись 14/16/20px вразнобой. `iconOnly` выводится сам (`!children && !!Icon`) и требует `aria-label`.

**Loading: «занят» ≠ «выключен».**

- Подпись **остаётся в потоке** и держит ширину (`opacity-0`), спиннер ложится **абсолютом по центру**. Кнопка не меняет ширину ни на пиксель — мишень не уезжает из-под курсора в момент клика. Вставлять спиннер перед подписью нельзя: он раздвигает её и расширяет кнопку.
- **Цвет остаётся полным.** Гаснет (`opacity-50`) только настоящий `isDisabled`. Поэтому гашение намеренно **не висит на `disabled:`-варианте**: во время загрузки кнопка тоже `disabled`, и иначе она неизбежно гасла бы, читаясь как недоступная.
- `aria-busy` на кнопке/ссылке, `cursor-wait`, клики заблокированы.

**Никаких наслоений снаружи.** Через `className` допустимы только позиционирование и раскладка (`absolute top-4 left-4`, `w-full`). Цвета, размеры, радиусы, иконки — только пропами. Нужного варианта нет → завести тему или ступень в компоненте, а не собирать её классами на месте (так снаружи вырос целый «призрачный деструктив», ставший `destructiveGhost`).

Содержимое кнопки обёрнуто в `span` (опора для loading-оверлея), поэтому иконка **не прямой потомок**: в call-site-селекторах использовать `[&_svg]`, а не `[&>svg]`.

### 4.3 Лестница контролов, поля, аватарки, сегмент-контрол

**Контролы живут на одной лестнице.** `Button`, `Input`, `Textarea` используют одни и те же ступени и радиусы — иначе поле и кнопка одной буквы не садятся по одной линии в общей строке. Единый набор `sm/md/lg/xl` (радиусы — токены `--radius-control-*`):

| Ступень | Высота | Радиус |
|---|---|---|
| `sm` | 32 | 10 |
| `md` | 36 | 10 |
| `lg` | 40 | 12 |
| `xl` | 48 | 14 |

#### Поля (`Input`, `Textarea`) — md/lg/xl

32px полям не отдаём (кегль ≥16px не влезает комфортно); дефолт поля — `lg` (40, r12, 16px).

| size | Высота | Кегль | Паддинг | Радиус | Иконка префикса |
|---|---|---|---|---|---|
| `md` | 36 | 16 | 12 | 10 | 16 |
| `lg` | 40 | 16 | 14 | 12 | 18 |
| `xl` | 48 | 16 | 16 | 14 | 20 |

**Кегль 16px на всех ступенях — это не эстетика, а баг-фикс.** Safari на iOS принудительно зумит страницу при фокусе на поле с кеглем меньше 16px. Apple решает это так же: у него текстовое поле 17pt независимо от высоты. Поэтому ступень поля меняет высоту, паддинг, радиус и иконку, но **никогда не кегль**. (Технически: поле — нативный `<input>`, кегль задаём явным `text-base` = 16px; гасить чужой `sm:text-sm`, как было у HeroUI `.input`, больше не нужно.)

#### `SegmentedControl` — вложенные радиусы выводятся, а не выбираются

У концентричных скруглений **внешний радиус = внутренний + отступ**. Иначе углы не параллельны и между пилюлей и треком видна щель переменной толщины. Раньше было `p-3px` + трек `r16` при пилюле `r10` — то есть 16 вместо положенных 13.

При отступе **2px** закон сходится со шкалой, и все четыре числа оказываются «своими»:

| size | Пилюля | Отступ | Трек | Радиус трека |
|---|---|---|---|---|
| `s` | 28, r8 | 2 | **32** | 8 + 2 = **10** |
| `m` | 32, r10 | 2 | **36** | 10 + 2 = **12** |

Высоты трека (32 и 36) попадают ровно в ступени `s` и `m` лестницы контролов.

#### `Avatar` — контентная шкала, но тот же закон радиуса

Аватарка не выстраивается с кнопкой кромка в кромку, поэтому у неё **своя** шкала — 32/40/48, и буквы `s/m/l` здесь означают не то же, что у контролов. Это единственное сознательное исключение, и оно записано именно чтобы не считаться ошибкой.

Радиус `square` при этом подчиняется общему закону и растёт с размером — **10 / 12 / 14**. Раньше на всех трёх размерах висел один `rounded-xl` (16px): отношение радиуса к стороне выходило 0.50 / 0.40 / 0.33, и аватарка 32px превращалась почти в круг — при том что `square` существует ровно затем, чтобы отличаться от `circle`.

**Связка с логотипом:** `LogoTile` в рейле стоит вплотную к аватаркам сообществ (48px), поэтому её радиус жёстко привязан к `Avatar size="l"` — 14px. Меняется один — меняется и второй, иначе в рейле углы разъезжаются.

#### Самодельные поля обязаны сидеть на той же лестнице

Часть полей собрана классами на месте (композер чата, комментарии, витрина, инвайт-ссылка) — обычно потому, что `Input`/`Textarea` требуют react-hook-form-контекст, а там локальный стейт. Это допустимо, но **спека — та же**: высота ступени (36/40/48), радиус ступени (10/12/14), редактируемый текст **16px**, фокус `inset-ring-2 primary-500`. Аудитом сняты: радиусы 16px и `text-sm` на редактируемом тексте — 14px на поле означает принудительный зум Safari на iOS при каждом фокусе.

У составных полей действует закон концентричности (§4.3/сегменты): кнопка внутри поля инвайта — внешний 14 − зазор 6 = **8**. Встроенные кнопки самого `Input` (крестик очистки, действие справа) — **ghost-прямоугольники**, радиус по тому же закону, радиус поля − зазор до кромки: `m` 10−4=**6**, `l` 12−6=**6**, `xl` 14−6=**8**. Фон — только на ховере.

**Как и у кнопок, классов поверх `Input`/`Textarea` в местах использования не бывает.** Иконки — пропами `Icon`/`IconRight` (принимают lucide-компонент или готовый элемент вроде `next/image`; размер навязывает слот ступени). Разделителя после иконки нет — иконка, зазор, текст, как в поле поиска Apple. Очистка — `isClearable` (появляется при непустом значении, клик не уводит фокус из поля — поведение iOS text field). Моно-значение — проп `mono`, не `className`.

Сознательные исключения — записаны, чтобы не считаться ошибками:
- **OTP-ячейки** (`ConfirmationForm`): 48×56, моно 2xl, r14 — специальный контрол, в лестницу не входит.
- **Edit-in-place микро-меток** (`LessonList`, переименование модуля): кегль наследуется от метки (11px uppercase), иначе строка разваливается. Это единственное редактируемое поле младше 16px; путь редкий и десктопный.

## 5. Layout

- Каркас: rail 60px (gray-100) → сайдбар сообщества 256px (белый, борт gray-200) → контент; `<768px` — Discord-drawer (85vw, max 340) + топ-бар с бургером.
- Радиусная система (лестничные токены): микроконтролы 8px · контролы/строки 10px (`--radius-control-sm/md`) · lg-контролы 12px (`--radius-control-lg`) · поля/плитки xl 14px (`--radius-control-xl`) · карточки 16px (`--radius-card`) · модалки/логин 24px (`--radius-modal`) · пиллы full. Произвольные значения вне шкалы не вводить.
- **Осторожно с утилитами радиуса:** в `@theme` переопределены токены, поэтому `rounded-md` = 12px, `rounded-lg` = **14px**, `rounded-xl` = **16px** (а не как в дефолтном Tailwind). Поэтому контролы (`Button`, `Input`, `Textarea`) берут радиус из **лестничных токенов** `rounded-(--radius-control-*)`, а не из `rounded-md/lg/xl`: ступень не должна съезжать оттого, что кто-то поменял общий токен. `Avatar`/`SegmentedControl`/`LogoTile` пока на явных литералах — точный перенос в токены см. Phase 3.
- Сетка 4px; карточки `p-5`; контентные колонки: чат full-width, посты 720px, урок 768px (`max-w-3xl`), настройки `max-w-2xl`, витрина 1000px.
- Тени: карточки `shadow-sm`, поповеры `shadow-lg`; глубина в основном рамками и слоями серого. Модалки — `shadow-modal`, слоистая (hairline-обводка + контактная + две амбиентные): одной тени не хватает — широкая размывается в грязь, короткая липнет к краю. В тёмной теме тень почти не читается, поэтому глубину держат приподнятая поверхность, hairline-обводка и внутренний блик сверху (`inset 0 1px 0 white/6%` — панели macOS ловят свет краем).
- z-слои: семантическая шкала в `@theme` (`--z-sticky/modal-backdrop/modal/dropdown/toast/tooltip` = 20…70); порталы (HeroUI — Dialog/Drawer/Toast; RAC — Popover/Tooltip) берут её через `z-[var(--z-modal)]` и т.п.; девятисотых значений нет.

## 6. Motion

Токены в globals: `--ease-out-quart` (базовый выход/ховеры), `--ease-out-expo` (входы слоёв), `--ease-in-out-quart` (движение на экране), `--ease-drawer` (iOS-кривая). Длительности: пресс 100–160 · тултипы 125–150 · меню 150–200 · модалки 240/160 (выход быстрее входа) · контент ≤300; всё ≤300мс.

- Слои входят от триггера (origin-aware) с масштабом от ~0.95, никогда от 0. **Исключение — модалка**: она ни к чему не привязана, поэтому зумится из центра экрана.
- Модалку центрировать **только флексом на обёртке**, не через `translate(-50%,-50%)` на самой карточке: `tailwindcss-animate` подменяет `transform` целиком, и на время анимации коррекция обнуляется — карточка вылетает из угла. Компенсировать это через `slide-in-from-*-1/2` нельзя: дробные утилиты у плагина не генерируются (в бандле их нет, ошибки при этом тоже нет — стиль просто молча не применяется).
- Появление нового контента (сообщения/комментарии/посты) — `@starting-style` через `[data-live]`: история не мигает, вставки всплывают (+6px, 240мс).
- Спринги (`motion`) только там, где есть жест/физика: drawer (bounce 0, dur 0.35, свайп-закрытие), лайк-«пружинка» (CSS keyframes 350мс). Сегмент-контрол на RAC Tabs больше **не слайдит**: активная пилюля переключается по `data-[selected]` без анимации индикатора (RAC не даёт скользящего маркера из коробки).
- Чарты: бары растут с каскадом 28мс, линия прорисовывается (`pathLength`), стат-карты — stagger 60мс.
- `prefers-reduced-motion: reduce` глобально гасит движение (transition/animation ≈0); в motion-компонентах `useReducedMotion` переключает на короткий фейд.
- Не анимировать: клавиатурные/сверхчастые действия, layout-свойства (кроме прогресс-баров и grid-rows схлопывания категорий).

## 7. Do / Don't

**Do**: держать один синий акцент; тонкие рамки вместо теней; radius только из шкалы; skeleton вместо спиннера на составных экранах; пустые состояния с иконкой-плиткой, заголовком и действием; тач-цели ≥44px.

**Don't**: `grid` без явных колонок в контейнерах с ограниченной шириной (неявный трек = `auto` раздувается под min-content и распирает контейнер вбок — в модалках всегда `grid-cols-1`, т.е. `minmax(0,1fr)`); цветные side-stripe борта; градиентный текст; стеклянные карточки без причины; зебра в таблицах; `transition: all`; scale(0)-появления; ease-in на UI; произвольные hex вне токенов (лайм логотипа — исключение); эмодзи как иконки (только lucide); голый `text-(--var)` для кегля — tailwind-merge примет его за цвет и съест соседний `text-white`/`text-ink` (текст падает в чёрный); токены-размеры пишем `text-(length:--var)`.

## 8. Адаптив и мобильные паттерны

Мобильная версия — важнейшая часть внешнего вида (большинство участников приходят с телефона). Ориентир — Apple HIG; референсы структуры — Discord (каркас) и Whop (витрина/оплата). Раздел снят с кода на этапе 17.7; правила обязательны для новых экранов и компонентов.

### 8.1 Брейкпоинт-стратегия — три полосы, а не «телефон/десктоп»

Ошибка, которую чинил этап 17.7: вся адаптивность висела на одном `md` (768) — планшет получал полную десктопную раскладку в тесноте. Полосы:

| Полоса | Ширина | Что показываем |
|---|---|---|
| Телефон | `<md` (<768) | одноколоночный каркас, Discord-drawer, bottom-sheet модалки, липкие CTA |
| Планшет | `md–lg` (768–1023) | **не мини-десктоп**: живой сайдбар, но master-detail экраны (курс: список↔урок чередуются до `lg`, не три колонки) |
| Десктоп | `≥lg` (≥1024) | полная многоколоночная раскладка |

- **`md` — это «есть ли сайдбар/рейл», `lg` — «есть ли место на многоколоночность».** Если layout ставит рядом ещё одну фиксированную колонку (список уроков, sticky-CTA витрины), рубеж её появления — `lg`, а не `md`: 60+256+ещё_одна на 768px не оставляет контенту места. Проверено на курсе (P0 аудита).
- `sm` (640) — для перестроек **внутри** телефонной полосы (стек→ряд карточек видимости, пилл→иконка).
- Десктоп-first в разметке остаётся историческим; для нового кода — mobile-first (базовые классы = телефон, `md:`/`lg:` добавляют).

### 8.2 Safe-area (iPhone-чёлка и home-индикатор)

`viewport-fit=cover` включён в [app/layout.tsx](app/layout.tsx) (`export const viewport`). Без него `env(safe-area-inset-*)` всегда нулевые. Утилиты в [globals.css](app/globals.css): `pb-safe` (чистый inset), `pb-safe-N`/`pt-safe-N` (`max(N, inset)` — держат базовый отступ на десктопе/Android), `pl-safe`/`pr-safe`. **Любая прижатая к нижней кромке интерактивная полоса** (композер чата, липкий CTA, футер bottom-sheet, drawer) обязана нести `pb-safe*`, иначе на iPhone уходит под home-индикатор.

### 8.3 Тач-цели ≥44px

HIG-минимум 44×44. Визуальный размер иконки можно оставить мелким — расширяется **тап-область**:
- `touch-hit` (globals.css) — псевдоэлемент растягивает область клика до 44px без изменения вёрстки. Для мелких иконок-кнопок (send, «···», «+», карандаш/корзина, радио). Требует ≥8px зазора до соседней цели (иначе перекрытие).
- Для строк-ссылок (каналы, уроки, пункты списка) — `min-h-11 lg:min-h-0`: на таче строка выше, на десктопе прежняя плотность.

### 8.4 Семантика «назад» — иерархия, никогда «в другое место»

«Назад» ведёт **на предыдущий уровень**, а не туда, где по совпадению живёт резолвер. Антипример (чинили в 17.7): в настройках «назад» вёл на `/communities`, откуда серверный резолв уводил в последнее сообщество — пользователь «телепортировался». Правильно: `router.back()` c фолбэком, либо явная ссылка на родителя (секция настроек → `/settings`).

### 8.5 Секционные настройки — iOS-список (drill-down)

Для экранов «профиль + N разделов» на мобиле: корень — **экран-список** (аватар/шапка → сгруппированный список с шевронами `›` → деструктив отдельной группой), раздел открывается отдельным экраном с «‹ Родитель». Не горизонтальные пиллы (обрезаются краем, чужеродны ДС). Десктоп — постоянный сайдбар; корень редиректит в первую секцию клиентски (`matchMedia`, вьюпорт серверу неизвестен). Референс — `SettingsHome` + settings-layout.

### 8.6 Липкий CTA-бар (конверсия)

Когда главное действие (оплата/вступление) уезжает за скролл на длинной странице — липкий бар снизу с ценой и кнопкой, появляется по `IntersectionObserver` (порог ~0.5: «хвост» карточки без кнопки не считается видимым). `pb-safe`, вход по `--ease-out-expo` ≤300мс, только `<lg` (на десктопе карточка sticky сама). Без выбранного варианта тап скроллит к карточке, а не оформляет вслепую. Референс — `StorefrontCtaBar`.

### 8.7 Bottom-sheet модалки (`<md`)

`Dialog` на телефоне — шит снизу на **HeroUI Drawer** (`placement="bottom"`): прижат к низу во всю ширину, верхние углы r24, `pb-safe`, вход/выход и **drag-to-dismiss встроены** в Drawer (та же кривая `--ease-drawer` = `cubic-bezier(0.32,0.72,0,1)`; ручной pointer-drag и кейфреймы `sheet-*` удалены). Ручка-хэндл (`Drawer.Handle`) показывается только там, где `dismissOnOutside`. Боковой Discord-drawer сообщества — тоже HeroUI Drawer (`placement="left"`); открытие свайпом-от-кромки осталось тонким кастомным триггером (у HeroUI такого нет). Раскладки кнопок футера (P3/P5 из §4.1) — мобильные, сохраняются.

### 8.8 Ряды кнопок и переполнение

- **Ряд из нескольких `size="l"` кнопок не помещается на телефоне.** Перестраивать, а не сжимать: главное действие full-width сверху + `grid-cols-2` для пары (курс), либо стек (`flex-col` до `lg`). Схлопывание кнопки до голой иконки при нехватке места (текст исчезает) — баг, не «адаптив».
- **Дублирующие действия убирать на мобиле** (в PostComposer пилл и кнопка «Опубликовать» делали одно и то же — кнопка распирала строку; на мобиле осталась одна).
- В flex-строке текст всегда `min-w-0` + `truncate`; соседний нетекстовый элемент — `shrink-0` + `whitespace-nowrap`, иначе он сжимает текст (имя автора → пара букв).
- Страница **никогда не скроллится горизонтально**. Широкий контент (таблицы, карусель, пиллы) — в своём `overflow-x-auto`, не распирает body. Служебные полосы на телефоне беречь: не больше одной над контентом (в курсе объединили «К урокам» + сегмент-контроль в одну).

### 8.9 Поля ввода — 16px (см. §4.3)

Редактируемый текст всегда ≥16px: Safari на iOS зумит вьюпорт при фокусе на поле мельче. Это уже зафиксировано в §4.3 (лестница контролов) — здесь как напоминание для самодельных полей.

### 8.10 Свайп-жесты

Открытие мобильного drawer — свайп от левой кромки (`≤24px`, горизонталь > вертикали ×1.5, чтобы не путать с вертикальным скроллом); закрытие — свайп влево по панели. Жесты живут на контейнере/панели, не на оверлейной полосе поверх контента (та крала бы тапы). Реализация — `CommunityShell`.

> Проверка адаптива — прод-сборкой (`pnpm build && pnpm start`) на ширинах 320/390/768/1024 × light/dark, а не эмуляцией в вёрстке. Каталог находок и метод скриншот-тура — `docs/mobile-audit.md`.

## 9. Токены — три тира

Токены разложены на три уровня (по Fluent 2 / Polaris / W3C DTCG); объявлены в `app/globals.css` с комментариями-маркерами. **Компоненты читают только Tier 2/Tier 3, никогда Tier 1 напрямую.**

- **Tier 1 — Primitive (сырые шкалы):** `--gray-*`, `--primary-*`, рампы радиуса/спейса/типа. Без семантики — просто значения-опции.
- **Tier 2 — Semantic (алиасы-решения):** `--color-surface`, `--color-ink`, `--color-border`, `--color-fill`, `--danger`, `--radius-control-*`, `--text-btn-*`. Ссылаются на Tier 1 и **переворачиваются в `.dark`** (и на тенанта).
- **Tier 3 — Component (опц.):** точечные переменные компонента (`--button-bg` и т.п.) — теперь **наши** (раньше их задавал HeroUI). Ссылаются на Tier 2.

**White-label = переопределение Tier 2 (+Tier 3)** в скоупе `[data-tenant="x"]`/`.theme-x`: ноль правок компонентов — это и есть механизм мультибренда. Тёмная тема (§2) — тот же приём: переворот Tier 2, а не `dark:`-утилиты.

## 10. Уровни компонентов (Atomic Design ↔ FSD)

Atomic-уровни ложатся на существующий FSD — **отдельной таксономии не заводим**:

| Atomic | FSD-слой | Примеры |
|---|---|---|
| Atoms / Foundation | `shared/components` | Button, Input, Textarea, Switch, Avatar, Skeleton, Separator, Logo |
| Molecules / Composite | `shared/components` | Dropdown/Menu, Dialog/Sheet, Tooltip, Tabs, SegmentedControl, Toast, Field |
| Organisms | `widgets` | Sidebar, Navbar, ProfileButton, формы/модалки с логикой |
| Templates / Pages | `pages` / `app` | экраны, лэйауты |
| Business | `features` / `entities` UI | доменные карточки и потоки |

Зависимости — по FSD: `app`/`pages` → `widgets` → `features` → `entities` → `shared`, импорт только через `index.ts` слайса. Оба движка (§4) невидимы выше `shared/components`.
